# Session Log: SDK Migration — Ink TUI, Auto-Dispatch Loop, Overstory Comparison

## Summary

Continued the SDK migration branch with three significant enhancements: replaced raw ANSI terminal rendering with an Ink (React-based) TUI for the watch mode, added an auto-dispatch loop so `foreman run` continuously dispatches new batches as agents complete, and redesigned the watch UI with rich per-agent cards showing model, cost, tool breakdown bar charts, and file lists. Also cloned and analyzed the overstory project for comparison with foreman.

## Key Decisions

### 1. Chose Ink over other TUI libraries
**Rationale:** Ink provides React component model for terminal rendering — handles cursor management, redraws, and layout automatically. Eliminates the manual ANSI escape code math that caused scrolling bugs. Selected over blessed (overkill/unmaintained) and terminal-kit (lower-level). Uses `React.createElement` directly to avoid JSX transpilation config changes.

### 2. Auto-dispatch loop instead of one-shot dispatch
**Rationale:** After agents complete a batch, foreman now automatically checks `bd ready` for newly unblocked tasks and dispatches the next batch. This replaces the previous behavior where users had to manually re-run `foreman run` after each batch. The loop exits when no new tasks are dispatched. `--no-watch` mode still exits after first dispatch.

### 3. Rich agent cards with tool breakdown bar charts
**Rationale:** User requested better visibility into agent progress. Each agent card now shows: animated spinner (running), model name, running cost, turn count, tool calls with mini ASCII bar chart of top-5 tools, files modified (with names), and log path for failures.

### 4. Overstory comparison for architectural insights
**Rationale:** User wanted to understand how overstory approaches the same multi-agent orchestration problem. Key takeaways: overstory has agent specialization (8 types), inter-agent mail system, tiered merge resolution, and multi-runtime support — but foreman has advantages in SDK-based spawning, live progress tracking, and session resume.

## Problems & Resolutions

### 1. TypeScript error: missing React type declarations
**Problem:** `TS7016: Could not find a declaration file for module 'react'`
**Resolution:** `npm install --save-dev @types/react`

### 2. JSX config not needed
**Problem:** Ink normally uses JSX/TSX which would require tsconfig changes
**Resolution:** Used `React.createElement` (aliased as `h`) directly, keeping tsconfig clean

### 3. Flaky ws.test.ts EADDRINUSE
**Problem:** WebSocket test failed due to port 19876 already in use
**Resolution:** Unrelated to changes — pre-existing flaky test. All relevant tests (cli, orchestrator, lib) pass.

### 4. Agents not picking up remaining work after batch completion
**Problem:** `foreman run` dispatched a batch, watched until done, then exited. Newly unblocked tasks required manual re-run.
**Resolution:** Added while loop in run command: dispatch → watch → check for more → dispatch again. Exits when no new tasks dispatched.

## Architecture Notes

### Watch UI Component Tree
```
WatchApp
  ├── Header ("Foreman — agent monitor")
  ├── Rule (horizontal separator)
  ├── AgentCard[] (one per run)
  │   ├── Header (spinner/icon + bead ID + status + elapsed)
  │   ├── Field (Model)
  │   ├── Field (Cost)
  │   ├── Field (Turns)
  │   ├── Tools + ToolBreakdown (bar chart)
  │   └── FilesChanged (file list)
  ├── Rule
  ├── SummaryBar (agents, tools, files, cost)
  └── DoneBanner (completion summary)
```

### Auto-Dispatch Loop Flow
```
while (true) {
  dispatch batch → print results
  if nothing dispatched or dry-run → break
  if watch mode → watchRunsInk() → loop back
  if no-watch → break
}
```

## Overstory vs Foreman Comparison

Key architectural differences documented:
- Overstory: 8 agent types, hierarchical spawning, inter-agent SQLite mail, tmux-based, 4-tier merge resolution, multi-runtime
- Foreman: flat worker model, in-process SDK, live progress tracking, session resume, auto-dispatch loop
- Potential borrowing: agent specialization, inter-agent comms, tiered merge resolution
